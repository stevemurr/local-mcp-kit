# Integrating LocalMCPKit

## Document status

This is the V1 host-app integration contract for the implemented package. It covers both deterministic in-memory composition and the production numeric-loopback HTTP, LocalOnly Bonjour, explicit pairing, and Keychain backends.

LocalMCPKit supports macOS 13 or later and Swift language mode 6. It ships focused library products, not an umbrella product.

## Choose products deliberately

| Integration | Products to import |
| --- | --- |
| Shared DTO/schema library | `LocalMCPContracts` |
| Platform-neutral discovery consumer | `LocalMCPContracts`, `LocalMCPDiscovery`, `LocalMCPConsumer` |
| macOS automatic discovery consumer | Above plus `LocalMCPDiscoveryBonjour` |
| Production in-process producer | `LocalMCPContracts`, `LocalMCPDiscoveryBonjour`, `LocalMCPProducer` |
| Tests and previews | Relevant products plus `LocalMCPTesting` |

`LocalMCPMCPAdapter` is internal and cannot be imported. Host code never needs an MCP wire or HTTP framework type.

V1 has no external package dependencies. The official MCP Swift SDK 0.12.1 was evaluated but not adopted because it requires Swift tools 6.1 and declares a moving DocC dependency. LocalMCPKit's isolated internal adapter implements the ratified MCP 2025-11-25 behavior while preserving the package's Swift tools 6.0/macOS 13 floor.

## Producer integration

A producer integration supplies five things:

1. A stable app identity.
2. Explicit command schemas and async handlers.
3. A producer-side pairing approval experience.
4. Lifecycle ownership and a secure grant store.
5. Operator controls for status and revocation.

### 1. Define identity

Choose a lowercase reverse-DNS stable ID once and keep it across releases:

```swift
let identity = ProducerIdentity(
    stableID: "com.example.notes",
    displayName: "Notes",
    version: "1.0.0"
)
```

The stable ID is the grant namespace. Changing it intentionally makes existing consumers see a different producer and requires new pairing. Display name and version may change without migrating grants.

The producer instance ID is generated by LocalMCPKit for each process launch. Do not persist or supply it as stable identity.

### 2. Define wire DTOs and schemas

Commands exchange app-neutral DTOs made of stable JSON-compatible primitives. Do not export SwiftUI/AppKit, database, SDK, or host model types.

For example:

```swift
struct SearchNotesInput: Codable, Sendable {
    let query: String
    let limit: Int
}

struct SearchNotesOutput: Codable, Sendable {
    let matches: [Match]

    struct Match: Codable, Sendable {
        let id: String
        let title: String
        let modifiedAt: String?
    }
}
```

Provide explicit JSON input and output schemas whose roots declare `"type": "object"`, as required by MCP tools. Registration rejects malformed or unsupported assertion schemas, and dispatch validates supported type, object, array, bounded string, numeric, combinator, constant, enum, and nullable assertions before the typed handler runs. Schema traversal has both depth and total-work budgets, including through local references. V1 deliberately rejects regular-expression `pattern`, `$dynamicRef`, and `$recursiveRef` assertions because they cannot meet the same bounded evaluation model; enforce lexical rules with bounded app code in the handler. Structured content must be an object even when no output schema is declared; a non-error structured result is also validated against the output schema when present. Host handlers must still enforce app-specific semantic rules and authorization; schema validation is not a data-policy boundary.

Use a stable, namespaced command name such as `notes.search`. Once released, changing a name or changing the meaning/type of a field is breaking. Add optional fields with defaults or introduce a new command/version for breaking behavior.

Annotations communicate expected behavior to a consumer:

```swift
let definition = CommandDefinition(
    name: "notes.search",
    title: "Search notes",
    description: "Search note metadata available to the current app session.",
    inputSchema: SearchNotesInput.schema,
    outputSchema: SearchNotesOutput.schema,
    annotations: CommandAnnotations(
        readOnly: true,
        idempotent: true,
        destructive: false,
        openWorld: false
    )
)
```

Annotations are hints, not permissions. LocalMCPKit still authenticates every call and the handler still enforces host-app data policy.

When annotations are omitted, MCP's conservative defaults apply: a tool is
treated as potentially destructive and open-world, while read-only and
idempotent default to false. Set every hint explicitly when the host can make a
stronger claim; never rely on an omitted field to make a tool appear safer.

### 3. Register handlers

The intended producer usage is concurrency-native:

```swift
let discovery = BonjourLocalMCPDiscovery()
let producer = LocalMCPProducer(
    identity: identity,
    configuration: .localOnly(),
    transport: LocalMCPHTTPProducerTransport(),
    advertiser: discovery,
    grantStore: try KeychainProducerGrantStore(),
    approval: approvalController,
)

try await producer.register(definition) {
    (input: SearchNotesInput, context: CommandContext) in

    try Task.checkCancellation()
    let matches = try await noteStore.search(
        query: input.query,
        limit: input.limit,
        deadline: context.deadline
    )
    let output = SearchNotesOutput(matches: matches.map(SearchNotesOutput.Match.init))
    return CommandResult.structured(output, text: "Found \(output.matches.count) notes.")
}
```

Handlers are `@Sendable` async work and are not automatically run on `MainActor`. Adapt UI-owned state through a small `Sendable` service/actor rather than routing command execution through view models or debounced UI services.

Use `CommandContext` for the sanitized grant/consumer identity, cancellation, deadline, and trace ID. It never exposes the bearer token. Check cancellation in long operations and apply narrower app-specific limits when appropriate.

Register commands before starting the producer. V1 freezes registration for the running lifetime; attempts to register while starting, running, or stopping fail without changing the registry. Duplicate names fail. Listing order is deterministic and must not depend on registration races.

### 4. Pairing approval

Provide an approval controller that can present one request and return Allow or Deny asynchronously. The UI shows:

- claimed consumer display name and stable ID;
- a recognizable suffix of its installation ID;
- the short verification code also shown in the consumer; and
- explicit Allow and Deny actions.

There is no default allow, background auto-approval, or â€śtrust every local appâ€ť mode. Expired/cancelled requests dismiss their prompt and cannot later be approved. The implemented network exchange is defined in [the local discovery profile](../Spec/local-discovery-v1.md#pairing-exchange). It is a LocalMCPKit bootstrap extension, not MCP OAuth.

Tests inject an in-memory approval controller, so they can pause, inspect, approve, deny, expire, replay, rate-limit, or cancel a pending request without UI or a real Keychain.

### 5. Compose lifecycle

The app owns the producer and starts it only after the services used by command handlers are ready:

```swift
try await producer.start()
```

Stop it before releasing those services or protected resources:

```swift
await producer.stop()
```

Start and stop are idempotent, but the host should still have one clear owner, normally an application dependency container. Do not create a producer in a SwiftUI view body or start it once per window. Observe producer state to drive operator UI.

If startup fails, present a sanitized error and leave the producer disabled/stopped. Do not add an emergency wildcard bind, unauthenticated mode, or manual advertisement workaround.

## Consumer integration

A consumer integration separates four concepts:

1. discovered availability;
2. descriptor compatibility;
3. producer-approved trust; and
4. consumer policy for exposing/calling tools.

### Define consumer identity

Use a stable app ID plus a random per-installation ID managed by the secure credential store:

```swift
let consumerIdentity = ConsumerIdentity(
    stableID: "com.example.assistant",
    displayName: "Example Assistant",
    version: "1.0.0",
    installationID: installationID
)
```

Do not derive installation identity from username, hardware, filesystem location, or a token. Tests inject a fixed value; real apps generate it once and store it in Keychain.

### Observe discovery

```swift
let discovery: any LocalMCPBrowsing = discoveryBackend

for await event in await discovery.events() {
    switch event {
    case .added(let instance):
        discovered[instance.instanceID] = instance
    case .updated(let instance):
        discovered[instance.instanceID] = instance
    case .removed(let instanceID):
        discovered.removeValue(forKey: instanceID)
    }
}
```

Discovery may populate a list, but do not label an item connected or trusted. Deduplicate by instance ID. Show incompatible descriptors separately from offline producers so users and diagnostics can distinguish version problems from availability.

Browsing is cancellable long-lived work. Own and cancel its task with the consumer feature lifecycle; do not create one browser per row or window.

The event stream is broadcast. A subscriber receives the current snapshot as deterministic `added` events before live changes; separate subscribers do not steal events from each other. Cancelling a feature's subscription does not implicitly cancel other consumers. A subscriber that cannot keep up is terminated rather than allowed to continue after a silent gap; resubscribe to receive a fresh snapshot.

### Pair explicitly

When the user selects Pair, create a consumer for the selected compatible instance and run the approval flow:

```swift
let client = LocalMCPConsumer(
    instance: selectedInstance,
    identity: consumerIdentity,
    connector: LocalMCPHTTPConnector(),
    grantStore: try KeychainConsumerGrantStore()
)

let grant = try await client.pair { verificationCode in
    verificationPresenter.show(verificationCode)
}
```

The consumer displays the short verification code while the producer asks for approval. Store the resulting credential only through the injected secure store. Pairing denial, expiry, cancellation, and producer unavailability are normal recoverable outcomes and should not be collapsed into a generic transport error.

Never copy a bearer credential into preferences, logs, crash annotations, command-line arguments, pasteboard, analytics, or a diagnostic bundle.

### List and call tools

```swift
let tools = try await client.listTools(grant: grant)

let result: SearchNotesOutput = try await client.call(
    "notes.search",
    input: SearchNotesInput(query: "roadmap", limit: 25),
    as: SearchNotesOutput.self,
    grant: grant
)
```

The consumer must initialize MCP before normal operations, negotiate an exact protocol version intersection, send `notifications/initialized` under the 2025-11-25 baseline, and send the grant on each request. The package wrapper owns those transport details and coalesces concurrent negotiation on one cached connection.

If a cached MCP session has expired, `listTools` may perform one fresh negotiation and retry the read-only listing. `call` never replays a command automatically: it re-establishes the session for a later explicit call but returns the original lifecycle error to the current caller. This matters even when a tool advertises idempotence, because annotations are hints and a response may have been lost after the producer performed work.

Pass a `deadline:` on UI-driven operations â€” `pair`, `initialize`, `listTools`, and `call` all accept one â€” so a wedged producer or network path surfaces as a timely `requestTimedOut` instead of an indefinitely suspended task. `requestTimedOut` is recoverable: the consumer stays usable and a later call (including a retried `pair`) may succeed.

Close a producer-bound consumer when its feature, account, or app-owned dependency container is torn down:

```swift
await client.close()
```

Closing cancels outstanding work, best-effort terminates the cached MCP session within a fixed bound, and makes later operations fail as cancelled. Updating or removing the selected producer instance invalidates the old route and session automatically. A successful re-pair also terminates the old session before the new grant becomes active.

Tool discovery does not mean every tool should be given to an LLM. The consumer app owns allowlists, user confirmation, model/tool exposure, command annotation interpretation, and any policy narrower than the producer grant.

On a producer process restart, a consumer may locate stored grant metadata by stable producer ID, but it must not treat the new advertisement as authenticated. LocalMCPKit deliberately never sends that bearer automatically after the instance changes. Require fresh explicit producer-side approval/rebinding, then use the newly returned grant. The host may retry transient connection failures with bounded cancellable backoff only while discovery still reports the instance; `LocalMCPConsumer` does not start background retries.

## Keychain and entitlements

Real integrations use `KeychainProducerGrantStore` and `KeychainConsumerGrantStore`. By default, each app stores its side in its own Keychain namespace with no explicit access group. Producer and consumer do not share Keychain items. Producer records contain credential digests rather than plaintext bearers; the consumer record contains only that consumer's own bearer and grant metadata.

If an app suite intentionally shares grants between its own signed components, pass an explicit access group that the host already entitles. LocalMCPKit does not infer or create one. Items are non-synchronizing and device-only; account migration or iCloud Keychain must not silently copy bearer grants to another Mac.

**Sandboxed apps must set `useDataProtectionKeychain: true`** on the store configuration. The default legacy file-based (login) keychain does not honor a keychain access group inside the App Sandbox, so `SecItem` calls fail with `errSecMissingEntitlement` and grants cannot be stored â€” pairing then fails right after user approval. Enabling the data-protection keychain requires the app to be signed by a team (not ad-hoc) and to carry a `keychain-access-groups` entitlement, for example `$(AppIdentifierPrefix)com.example.app`. The default stays `false` so non-sandboxed hosts keep using the login keychain unchanged.

```swift
grantStore: try KeychainProducerGrantStore(
    configuration: .init(useDataProtectionKeychain: true) // sandboxed apps
)
```

For sandboxed production networking:

- a producer app needs the macOS App Sandbox network-server entitlement;
- a consumer app needs the network-client entitlement; and
- an app that acts as both needs both.

Apply entitlements to the embedding app target, not this Swift package. Verify signed release behavior as well as debug builds. DNS-SD remains LocalOnly regardless of entitlement; entitlements are not a substitute for the explicit interface index or loopback bind.

## Required producer operator experience

A shipping producer provides at least:

- enable/disable;
- observable starting/running/stopping/stopped/error status;
- explicit pairing approval with identity and verification code;
- a grant list and immediate revoke;
- credential rotation or a clear revoke-and-pair-again path; and
- redacted diagnostics.

Show the stable producer ID and LocalOnly nature in diagnostics. The ephemeral port may be shown for troubleshooting, but it is not stable configuration and is not identity.

## Testing an integration

Import `LocalMCPTesting` and test host behavior without GUI, Keychain, Bonjour, or sockets. Every command should have tests for:

- schema acceptance and rejection at all boundaries;
- typed input decode and structured/text output encode;
- deterministic registration/listing and duplicate name rejection;
- successful handler behavior;
- host data errors mapped to sanitized command failures;
- timeout and cancellation propagation;
- unauthorized/revoked calls never entering the handler;
- sensitive seeded values absent from logs/errors; and
- any app-specific privacy or authorization rule.

Each producer integration should also run an in-memory end-to-end test:

```text
start â†’ discover â†’ pairing required â†’ approve â†’ initialize â†’ initialized
      â†’ list â†’ call â†’ revoke â†’ call rejected â†’ stop â†’ removed
```

Add a second flow with two consumer installations and prove revoking one does not affect the other. Use invocation counters to prove rejected calls did not dispatch. Use an injected clock/random source to test expiry and rotation deterministically; do not sleep or assert on real random values.

Add integration coverage for the embedding app's signed entitlements and lifecycle in addition to package tests. Exercise the real HTTP/Bonjour path in a separate-process test, while keeping Keychain unit tests on injected access doubles so a test never touches a developer's Keychain. A clean test run must leave no listener, discovery registration, task, prompt, rendezvous secret, or Keychain test item behind.

## Errors and user messaging

Handle stable error categories rather than matching framework strings:

| Category | Suggested app behavior |
| --- | --- |
| Incompatible discovery/MCP version | Show an update/compatibility state; do not retry rapidly. |
| Producer unavailable | Preserve selection and retry with bounded backoff while still discovered. |
| Pairing required | Offer Pair; do not invoke automatically. |
| Pairing denied/expired | Explain the outcome and let the user initiate a new request. |
| Unauthorized/revoked | Remove or quarantine the local credential and offer pairing again. |
| Invalid command input | Treat as a consumer/schema bug or invalid user input, not a retryable transport failure. |
| Command failed | Show sanitized producer context; log only an allowed trace ID/category. |
| Timed out/cancelled | Respect user cancellation and avoid unbounded automatic replay. |

Do not show raw HTTP, Keychain, DNS-SD, SDK, or listener error descriptions directly to users; those may contain implementation details. LocalMCPKit supplies redacted diagnostic context separately.

## Schema and compatibility changes

Descriptor readers ignore unknown fields, but required-field type/meaning changes require a new descriptor schema version. See [the V1 profile](../Spec/local-discovery-v1.md).

For commands:

- preserve released names and field meanings;
- add optional fields only when old consumers remain valid;
- do not repurpose enum cases;
- prefer a new command/version for breaking input or output;
- retain structured output plus useful text fallback; and
- fixture-test old and new encodings.

Changing a stable producer ID intentionally invalidates the grant namespace. Changing only app version or display name does not.

## Troubleshooting

### A producer does not appear

Check, in order:

1. Producer state reached running.
2. Its listener bound `127.0.0.1` and has a resolved nonzero port.
3. DNS-SD registration and browsing both use the LocalOnly interface.
4. The TXT record is V1-compatible.
5. The descriptor route returns a bounded compatible document on the same port.
6. App Sandbox network entitlements are present in the signed app.

Do not change the bind address or discovery interface to make a failing test pass.

### A producer appears but pairing fails

Confirm the user initiated pairing, both apps show the same verification code, the request has not exceeded 120 seconds, the producer has capacity under its pairing rate limits, and the approval callback is active. Logs must identify only a sanitized request/trace category, never the nonce/code.

### A stored grant stopped working

It may be revoked, rotated, expired by host policy, lost from Keychain, or namespaced under a changed producer/consumer identity. Remove the rejected local credential and offer explicit pairing again. Never fall back to unauthenticated MCP.

### Calls time out or cancel

Verify the host handler is not forced through UI state, observes `CommandContext` cancellation/deadline, and does not hold an actor while waiting on unrelated work. Preserve the trace ID and sanitized category for correlation.

## Adoption order

1. Prove command schemas, handlers, authorization, and lifecycle using `LocalMCPTesting`.
2. Compose `LocalMCPHTTPProducerTransport`, `BonjourLocalMCPDiscovery`, an explicit `PairingApproving` implementation, and `KeychainProducerGrantStore` in one app-owned dependency container.
3. Add enable/status, approval, grant list/revoke, and redacted diagnostics UI.
4. Apply the host target's sandbox entitlements and test the signed product, not only SwiftPM.
5. Add a separate-process negotiated lifecycle test and run UI tests in an isolated VM.

The sibling File Search integration is the first real app integration and is staged alongside V1 validation. Its `files.search` command is read-only and metadata-only. It queries `FileIndexDatabase` through a small `Sendable` boundary rather than the UI's `MainActor` service. Policy is fixed as follows:

- `query` may be empty or whitespace-only, in which case the existing database policy returns the most recently modified indexed items; nonempty queries are capped at 512 characters;
- `limit` defaults to 25 and has a hard maximum of 100;
- `scope` is one of `all_indexed`, `home`, or `icloud`; and
- results expose metadata onlyâ€”no contents, open/reveal, mutation, rebuild, or arbitrary-path operation.

The generic package contains no File Search types. The app integration owns its privacy-preserving result DTO, approval/operator UI, network-server entitlement, and lifecycle ordering: start after index access is ready and stop MCP before releasing security-scoped folder access.
